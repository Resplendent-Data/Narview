# Narview V1

## Problem Statement

Reviewers who work through GitHub pull requests with many CodeRabbit and human review threads lose track of what they have actually inspected. GitHub can show resolved and unresolved conversations, but it does not give the reviewer a clean per-user attention layer for "I looked at this," especially when AI tools or CodeRabbit resolve threads automatically after new commits.

Large pull requests make this worse. A reviewer may need to move quickly through dozens of review threads, thousands of changed lines, or hundreds of files, but GitHub's file-first review flow makes it hard to see the important work, resume exactly where they left off, and separate local review attention from GitHub's remote conversation state.

Narview exists to make high-comment pull requests easier to clear with confidence.

## Solution

Narview is a macOS and Linux desktop app for reviewing one GitHub pull request at a time. It connects to github.com through OAuth, reads pull request metadata, file changes, review threads, checks, and GitHub-provided descriptions, then presents the pull request through a Guided Review Workspace.

The app is remote-first: users do not need a local clone or local Git installation. Narview fetches GitHub data incrementally, caches it locally, and lets users quickly switch between pull requests from saved Workspace repositories or open a specific pull request by URL.

The primary workflow is queue-driven review. Review Queues surface CodeRabbit and human Review Threads with filters for Reviewed, Resolved, Outdated, author, and other states. Users can mark Review Threads as Reviewed locally, reply to existing GitHub Review Threads, resolve or unresolve threads on GitHub, perform confirmed Bulk Actions, and resume a Review Session exactly where they left off.

Narview also provides a Review Overview with deterministic Hotspots, merge readiness context, checks summary, file-level Viewed state, lazy syntax-highlighted diffs, expandable surrounding context, full-file view, keyboard-first navigation, and Handoff Packets for external coding agents. Narview itself remains LLM-free in v1.

## User Stories

1. As a reviewer, I want to sign in to github.com with OAuth, so that I can access pull requests without manually creating a token.
2. As a reviewer, I want Narview to store my OAuth token in OS secure storage, so that credentials are not written into cache files.
3. As a reviewer, I want to save GitHub repositories into a Workspace, so that Narview shows only the pull requests I care about.
4. As a reviewer, I want to open a pull request directly by GitHub URL, so that I can review an ad hoc pull request without changing my Workspace first.
5. As a reviewer, I want to see open non-draft pull requests from saved repositories by default, so that my active review work is easy to find.
6. As a reviewer, I want to include draft pull requests through a filter, so that I can inspect work in progress when needed.
7. As a reviewer, I want fast switching between pull requests, so that I can move through many reviews without maintaining local checkouts.
8. As a reviewer, I want Narview to remember the last active queue, filter, thread, file, mode, and position in each pull request, so that I can resume where I left off.
9. As a reviewer, I want a Review Overview for each pull request, so that I can orient myself before diving into threads or files.
10. As a reviewer, I want deterministic Hotspots based on visible pull request signals, so that important areas are highlighted without relying on an LLM.
11. As a reviewer, I want hotspot reasoning to be explainable, so that I can trust why a file or area appears early.
12. As a reviewer, I want to review one pull request at a time, so that the workspace stays focused and does not drift into stack management.
13. As a reviewer, I want a Guided Review Workspace with a review map, review canvas, and inspector, so that I can see navigation, code, and thread actions together.
14. As a reviewer, I want focus mode, so that I can hide surrounding panels during deep diff reading.
15. As a reviewer, I want Review Queues for CodeRabbit Threads and human Review Threads, so that I can clear discussion work systematically.
16. As a reviewer, I want all CodeRabbit Threads available with filters, not only unresolved ones, so that auto-resolved but personally unreviewed threads are still visible.
17. As a reviewer, I want Reviewed to be my local per-user attention state, so that it records what I personally looked at.
18. As a reviewer, I want Reviewed to be separate from GitHub's Resolved state, so that local attention and remote conversation state do not get confused.
19. As a reviewer, I want Review Threads keyed by GitHub review thread ID, so that local Reviewed state follows the thread even if diff positions move.
20. As a reviewer, I want recovery context stored for Review Threads, so that Narview can diagnose or recover from missing or changed GitHub thread data.
21. As a reviewer, I want Outdated Review Threads to remain visible but clearly marked, so that I know when a discussion belongs to an older diff.
22. As a reviewer, I want to mark Review Threads as Reviewed or unreviewed, so that my personal review queue reflects my attention.
23. As a reviewer, I want resolving a Review Thread inside Narview to also mark it Reviewed locally, so that strong review actions count as inspection.
24. As a reviewer, I want unresolving a Review Thread not to remove Reviewed state, so that reopening a discussion does not erase my review history.
25. As a reviewer, I want to reply to existing GitHub Review Threads, so that I can keep the thread workflow inside Narview.
26. As a reviewer, I want to resolve and unresolve Review Threads in Narview, so that I do not need to jump to GitHub for common thread actions.
27. As a reviewer, I want confirmed Bulk Actions for local state and GitHub thread state, so that I can move quickly without accidental remote writes.
28. As a reviewer, I want Bulk Actions that write to GitHub to require explicit confirmation, so that resolving or unresolving many threads is intentional.
29. As a reviewer, I want file-level Viewed state separate from thread-level Reviewed state, so that looking at a file does not imply handling every discussion.
30. As a reviewer, I want marking a File Change as Viewed not to clear related Review Threads, so that review attention remains precise.
31. As a reviewer, I want both unified and side-by-side diff views, so that I can choose the format that fits the change.
32. As a reviewer, I want broad multi-language syntax highlighting, so that diffs remain readable across repositories.
33. As a reviewer, I want syntax highlighting to be lazy and limited to visible content, so that huge pull requests stay responsive.
34. As a reviewer, I want surrounding diff context and full-file view, so that I can understand a thread without needing a local clone.
35. As a reviewer, I want full-file fetching to be on demand, so that Narview does not download every file before review can start.
36. As a reviewer, I want binary and non-text file changes to be visible as changed files, so that the Review Overview is complete even without rich previews.
37. As a reviewer, I want checks summarized with links to details, so that I can understand CI context without Narview becoming a log viewer.
38. As a reviewer, I want Merge Readiness context, so that I can see remaining blockers without merging from Narview.
39. As a reviewer, I want a visible Keyboard Flow, so that I can move through high-volume review work mostly from the keyboard.
40. As a reviewer, I want keyboard shortcuts to be obvious in the UI, so that I can discover actions without reading docs.
41. As a reviewer, I want a command palette, so that I can find navigation and review actions quickly.
42. As a reviewer, I want Handoff Packets for selected Review Threads and context, so that I can pass focused work to an external coding agent.
43. As a reviewer, I want Handoff Packets exported as Markdown, so that I can paste them into external AI tools.
44. As an integration developer, I want Handoff Packets built from a structured internal model, so that future integrations can preserve IDs and intent.
45. As a reviewer, I want Narview to avoid LLM calls, so that PR review stays fast, private, explainable, and cost-free.
46. As a reviewer, I want limited offline read support for cached data, so that I can continue reading loaded pull request content when the network drops.
47. As a reviewer, I want GitHub writes to require network access, so that Narview does not create a fragile offline mutation queue.
48. As a reviewer, I want cached pull request data bounded by size and recency, so that Narview does not consume unbounded disk space.
49. As a reviewer, I want to pin pull requests, so that selected review data remains available longer.
50. As a reviewer, I want normal cache clearing to preserve Reviewed, Viewed, and Review Session state, so that freeing disk space does not erase my work.
51. As a reviewer, I want a separate reset action for local review history, so that forgetting review state is deliberate.
52. As a reviewer, I want rate-limit status to be visible when it affects data or actions, so that missing or delayed GitHub data is understandable.
53. As a reviewer, I want restrained refresh on open, focus, manual refresh, and active pull request background refresh, so that data stays fresh without burning API budget.
54. As a reviewer, I want no telemetry in v1, so that private code and review metadata do not leave my machine unexpectedly.
55. As a reviewer, I want local diagnostics export to be explicit and previewable, so that I control what debugging information leaves my machine.
56. As a maintainer, I want auto-updates delivered through GitHub Releases, so that users receive fixes without manual reinstall work.
57. As a maintainer, I want release artifacts signed for updates, so that update delivery is trusted.
58. As a macOS user, I want builds signed and notarized, so that the app launches without avoidable platform warnings.

## Implementation Decisions

- Narview v1 targets macOS and Linux desktop only.
- Narview is a Tauri v2 desktop app with a React and TypeScript frontend backed by a Rust core.
- The frontend uses shadcn/ui and Radix primitives as a component foundation.
- Tailwind CSS is used with shadcn/ui for v1 styling.
- The app supports light and dark themes from the start, defaulting to system preference.
- The visual density should be compact and desktop-oriented, closer to Linear-like density than GitHub's roomier web layout.
- Narview uses one main window in v1.
- Narview reviews pull requests from remote GitHub data and does not require a local clone.
- Narview does not require Git to be installed locally in v1.
- Narview supports github.com only in v1, not GitHub Enterprise Server.
- Narview uses OAuth-only GitHub authentication in v1.
- OAuth tokens are stored in OS secure storage: macOS Keychain and Linux Secret Service or an equivalent secure storage provider.
- If secure token storage is unavailable, Narview should fail clearly instead of storing tokens in plain files.
- Narview supports one signed-in GitHub account in v1.
- The Workspace is a local saved set of GitHub repositories.
- Narview lists open non-draft pull requests from saved Workspace repositories by default.
- Draft pull requests are available through a filter.
- Closed and merged pull requests are outside the normal active list but may be opened by URL if GitHub data is available.
- Pull request URL quick-open is supported.
- Review is focused on one Pull Request at a time.
- The primary surface is the Guided Review Workspace, not a GitHub-style files-changed page.
- The Guided Review Workspace has a left review map, center review canvas, right inspector, and focus mode.
- Review ordering defaults to Review Queues, Hotspots, and Review Paths rather than alphabetical file order.
- Narview does not depend on CodeRabbit Change Stack.
- Review Overview is the first orientation surface for a Pull Request.
- Hotspots are deterministic and explainable in v1.
- Hotspot inputs may include changed line count, file type, Review Thread density, CodeRabbit metadata if exposed by GitHub content, check context where inferable, generated/vendor suppression, and new/renamed/deleted file weighting.
- Narview does not use LLM calls for Review Overview, Hotspots, or summaries.
- Narview uses GitHub-provided title, description, metadata, Review Threads, Checks, and deterministic analysis for summaries.
- Repository settings support global hotspot defaults plus per-repository overrides.
- Repository overrides are local settings for v1; checked-in `.narview.yml` is deferred.
- Narview fetches pull request metadata, review threads, and file summaries before fetching full diff content.
- Diff content is paged and cached on demand.
- Diffs are GitHub-compatible as the source of truth and enriched internally for rendering, virtualization, hotspot scoring, and local state.
- Review Thread identity is keyed by GitHub review thread GraphQL ID.
- Review Thread recovery context includes repository, pull request number, path, line context when available, author, and creation metadata.
- Reviewed is Narview-local, per-user state for Review Threads.
- Viewed is Narview-local, per-user state for File Changes.
- Review Session stores resume position and active context per Pull Request.
- Review Session is continuity state, not proof of review attention.
- Resolving a Review Thread in Narview also marks it Reviewed locally.
- Unresolving a Review Thread in Narview does not remove Reviewed state.
- Marking a File Change as Viewed does not mark related Review Threads as Reviewed.
- All CodeRabbit Threads are ingested and filterable, not only unresolved threads.
- Human Review Threads are supported alongside CodeRabbit Threads.
- Outdated Review Threads remain visible and are clearly marked.
- Narview supports replying to existing Review Threads.
- Narview supports resolving and unresolving existing Review Threads.
- Narview does not create new Review Threads or new line comments in v1.
- Narview does not submit full GitHub reviews such as approve, request changes, or general review comments in v1.
- Bulk Actions can modify local review state and GitHub Review Thread state.
- Bulk Actions that write to GitHub require explicit confirmation.
- GitHub writes require network access; offline writes are not queued in v1.
- Narview supports limited offline read access to cached metadata, Review Threads, loaded diffs, Reviewed state, Viewed state, and Review Session state.
- Cached pull request metadata and diff content are stored plainly in the local app data directory in v1.
- Cached data is bounded by size and recency.
- Users can pin pull requests to keep cached data longer.
- Normal cache clearing removes fetched GitHub metadata and diff content but preserves Reviewed, Viewed, and Review Session state.
- Resetting local review history is a separate explicit action.
- Narview keeps local redacted operational logs only.
- Logs must not contain raw code, diff hunks, Review Thread bodies, OAuth tokens, or sensitive request details.
- Diagnostics export is explicit and previewable.
- Narview collects no telemetry or analytics in v1.
- GitHub data fetching is incremental, cache-aware, and rate-limit-aware.
- GitHub rate-limit status is visible when it delays data or actions.
- Refresh happens on pull request open, app focus, manual refresh, and restrained background refresh for the active Pull Request.
- Checks are shown as summary status with names, timing, and blocking indicators where available.
- Full CI log viewing is deferred; detailed checks link out to GitHub or the CI provider.
- Check annotations are not first-class inline review items in v1.
- Merge Readiness is visible, but Narview does not merge pull requests in v1.
- Diff views support unified and side-by-side modes with local preference memory.
- Syntax highlighting supports many languages and is lazy for visible or near-visible content.
- Users can expand surrounding diff context and fetch whole files on demand.
- Symbol navigation and command-click definition lookup are deferred beyond v1.
- Binary, image, and non-text file changes are represented as changed files but do not receive rich previews or specialized diffing in v1.
- Handoff Packets contain selected Review Threads, nearby diff hunks, file paths, pull request metadata, and explicit user intent.
- Handoff Packets are generated from a structured internal model.
- Handoff Packets can be exported as Markdown for external tools.
- Narview does not apply AI-generated code changes directly in v1.
- Keyboard Flow is first-class and visible in the UI.
- Expected keyboard actions include moving between threads, marking Reviewed, resolving, replying, switching filters, entering focus mode, opening files, and copying Handoff Packets.
- A command palette is included in v1 for navigation, review actions, filter switching, bulk actions, focus mode, handoff creation, and shortcut discovery.
- Narview ships with auto-update support in v1.
- Releases use GitHub Actions, SemVer tags, GitHub Releases, signed Tauri updater artifacts, and platform installers.
- macOS builds must be signed and notarized for public distribution.
- Linux ships AppImage first, with additional formats such as `.deb` added when needed.
- V1 success is review-thread throughput with confidence; Hotspots, diff rendering, and Merge Readiness support that outcome but are secondary.

## Testing Decisions

- Because the repository is new, there is no existing test prior art to preserve.
- Tests should prioritize external behavior and durable state transitions over component internals.
- Domain/state tests should cover Reviewed, Viewed, Resolved, Outdated, Review Session, Bulk Action, cache clearing, and pinned cache behavior.
- GitHub API integration tests should use mocked GitHub GraphQL/REST responses for pull request metadata, review threads, replies, resolve/unresolve mutations, checks, rate-limit responses, and error cases.
- Authentication tests should verify OAuth flow boundaries, token persistence through secure storage abstractions, sign-out, and secure-storage unavailable behavior.
- Cache tests should verify incremental fetching, bounded eviction, pinning, offline read behavior, and preservation of local review state during cache clearing.
- Diff renderer tests should cover unified and side-by-side modes, lazy hunk loading, lazy syntax highlighting, context expansion, full-file fetch, outdated thread anchors, huge files, generated/vendor suppression, binary file awareness, and non-text file fallback.
- Hotspot tests should verify deterministic scoring, repository override behavior, generated/vendor suppression, and explainable ranking output.
- Review Queue tests should cover filters for CodeRabbit Threads, human Review Threads, Reviewed, unreviewed, Resolved, unresolved, Outdated, and combinations such as Resolved plus unreviewed.
- GitHub write tests should verify reply, resolve, unresolve, confirmed Bulk Actions, partial failures, retryable failures, permission failures, and network failures.
- Keyboard Flow tests should verify that major review actions are reachable by keyboard and that visible shortcut cues match the active bindings.
- Command palette tests should verify discoverability and execution of navigation, filter, focus, review, bulk, and Handoff Packet commands.
- Review Session tests should verify restoring active queue, filter, thread, file, diff mode, focus mode, and position after app restart or pull request switch.
- Security/privacy tests should verify that tokens are never written to SQLite, logs, diagnostics, or cache files.
- Logging tests should verify redaction of raw code, diff hunks, thread bodies, tokens, and sensitive URLs.
- No-telemetry tests should verify that the app has no analytics emission paths in v1.
- Release tests should verify Tauri updater metadata, signed update artifacts, installer creation, macOS signing/notarization where CI secrets are available, and GitHub Releases publication.
- UI smoke tests should cover macOS and Linux where possible, including app launch, sign-in state, Workspace setup, PR list, Review Overview, Review Queue, diff rendering, thread actions, command palette, theme switching, and update check.
- Performance tests should use synthetic large pull request fixtures to verify that PR overview and Review Queues become usable before all diff content is loaded.

## Out of Scope

- Mobile apps.
- Browser-hosted app experience.
- Windows support in v1.
- GitHub Enterprise Server support.
- Multiple GitHub accounts.
- Local clone requirement.
- Local Git requirement.
- Git operations such as checkout, branch switching, rebasing, committing, or pushing.
- Managing stacked pull requests.
- Merging pull requests.
- Full GitHub review submission, including approve, request changes, or general review comments.
- Creating new Review Threads or new line comments.
- Full CI log viewer.
- First-class check annotations in diffs.
- LLM-generated summaries, LLM-generated hotspot ranking, or LLM analysis inside Narview.
- Direct AI code changes.
- Offline GitHub writes or queued remote mutations.
- Rich image, binary, notebook, or non-text diff previews.
- Symbol navigation, command-click definition lookup, and broader code intelligence.
- Checked-in `.narview.yml` configuration.
- Telemetry or analytics.
- Multi-window support.
- Team-synced Reviewed or Viewed state.
- Replacing GitHub, Graphite, CodeRabbit, or a CI provider.

## Further Notes

- The ADR filename for the OAuth decision still mentions device flow and PAT fallback, but the ADR title and content now reflect the final decision: OAuth-only GitHub authentication for v1.
- The current repository is primarily decision documentation. Implementation scaffolding still needs to be created.
- The first implementation slices should favor the product's north star: OAuth, Workspace setup, pull request listing, Review Overview, Review Queue, local Reviewed state, and basic thread actions.
- The first visual prototype should validate the Guided Review Workspace, compact shadcn/Tailwind styling, keyboard cues, command palette, and queue-first flow before investing heavily in secondary diff features.
- The first performance prototype should validate incremental GitHub fetching, virtualized Review Queues, and lazy diff rendering against synthetic large pull request data.
- Before public distribution, release engineering must include macOS signing/notarization, Tauri updater signing keys, GitHub Actions release automation, and GitHub Releases update metadata.
