# Success is review confidence over changed logic

Narview's **Attention Map** redesign should optimize for **Review Confidence**: a reviewer can explain and clear the changed logic in a pull request after inspecting relevant **Review Targets**, **Context Nodes**, **Review Threads**, and tests. Success should not be measured primarily by comment volume, approval speed, or merge velocity, because those can reward shallow review behavior.

Concrete acceptance signals include opening a pull request into a usable Attention Map after clone and indexing, understanding why top Review Targets are ordered first, starting line-level or file-level Review Threads, marking Review Targets and Review Threads Reviewed, copying a **Human Feedback Packet**, and seeing changed previously reviewed targets become **Needs Re-Review** after new commits.
